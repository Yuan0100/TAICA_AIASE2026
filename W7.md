## PART 0｜導論：為什麼從「基礎套件」看起？

- 市面已有成熟框架（OpenClaw、LangGraph、AutoGen、CrewAI），install 完可直接跑「多代理人系統」
- **危險**：你看到的只是「運作的結果」，而不是「運作的原理」
  - 出錯時不知道為什麼錯
  - 要客製化時不知道從哪裡改
- 框架原始碼為通用性高度抽象化 → 你永遠只是使用者，而不是工程師

### 代理人系統的三個核心本質

1. **Context Engineering**：餵給 LLM 看什麼、不給它看什麼 → 決定行為邊界
2. **Harness（確定性外殼）**：用確定性的 shell 包住 LLM 的機率性 core → 定義不可談判的規則
3. **Loop with Verification**：設計可被驗證的迴圈，讓代理人在其中自主迭代 → 清楚的 verification criterion 是自主優化的前提

> 如果你連自己系統的 verification loop 在哪裡都看不到，你就永遠只是這個系統的使用者。

## PART 1｜karpathy/autoresearch：Agentic Loop 的最小可運作原型

### 為什麼選這兩個套件？

| 套件 | 作者 | 特色 | 我們學什麼 |
|------|------|------|-----------|
| karpathy/autoresearch | Karpathy | 3 個核心檔案，約 100 行的 program.md | Agentic Loop 的最小骨架 |
| garrytan/gstack | Garry Tan (YC CEO) | 23 個 slash command，每個都是純 Markdown skill | Skill-Based Orchestration |

- 呼應 **Spec-Driven Development**：prompt 是 micro-spec，program.md / SKILL.md 是 macro-spec

### autoresearch 的三檔分工

| 檔案 | 角色 | 說明 |
|------|------|------|
| `prepare.py` | **Harness**（不可改動） | 固定常數、資料下載、BPE tokenizer、評估工具 |
| `train.py` | **Probabilistic Core**（Agent 改） | GPT model + Muon/AdamW optimizer + training loop |
| `program.md` | **Spec**（Human 改） | Agent 的指令書，設計研究組織的地方 |

- 關鍵設定：每次訓練固定 **5 分鐘 wall-clock**；評估指標：**val_bpb**（越低越好，與 vocab size 無關）

### Agent 的工作循環

1. 改 `train.py`
2. 跑 5 分鐘
3. 看 val_bpb 是否下降
4. 保留或捨棄 → 下一輪

### 為什麼這個設計是「神作」？

| autoresearch 元件 | AI Native 概念 | 意義 |
|-------------------|---------------|------|
| train.py | Probabilistic Core | LLM 可自由修改的區域，允許創造性發揮 |
| prepare.py | Deterministic Harness | 不給 Agent 碰，保證資料與評估一致 |
| val_bpb + 5-min budget | Verifiability Criterion | 清楚、可計算、跨實驗可比較的 loss function |
| program.md | Spec / Skill | 人類「程式設計」代理人組織的地方 |
| 通宵跑 100 次實驗 | Snowball / Incremental Loop | 每次迭代都有可執行交付物 |

### 它揭露了哪些「知其所以然」的細節？

1. **為什麼固定 5 分鐘？**
   - 固定時間預算讓所有實驗可直接比較
   - 用 wall-clock 當 normalizer → 控制變因的工程實踐

2. **為什麼用 val_bpb 而不是 val_loss？**
   - Agent 可能修改 vocab_size
   - Bits per byte 讓不同 tokenizer 設定下的實驗公平比較

3. **為什麼 prepare.py 不能被 Agent 修改？**
   - Harness 核心原則：若資料處理也可被 Agent 改，它可能「讓 eval set 變小」來作弊降低 val_bpb
   - Harness 的本質 = 定義什麼是「不可談判的規則」

4. **為什麼 program.md 故意保持簡陋？**
   - 這不是答案，而是可迭代的起點
   - 調整 program.md = 設計你自己的研究組織

**Verifiability Mindset + Harness Engineering 的經典案例：**
- 若用 per-token loss → Agent 會發現「把 vocab_size 改大就能降 loss」→ reward hacking
- 改用 val_bpb（byte-level normalization）→ Agent 就算改 vocab_size，也得真的讓模型變好才能贏

### 給研究生的「五步移植法」

| autoresearch 元件 | 你的研究任務對應項 |
|-------------------|------------------|
| train.py | 你的待優化模組（研究的「被動部分」） |
| val_bpb | 你研究任務的驗證準則 |
| program.md | 你寫給 Agent 的研究 SOP |
| 5-min budget | 你定義的終止條件 |
| prepare.py 防改 | 你的 Harness 邊界 |

> 核心問題：「我手上的研究任務，有哪一塊可以讓 Agent 通宵幫我跑？」這個問題的答案，決定了你跟同齡研究生的產出差距。

## PART 2｜garrytan/gstack：用 Markdown 組一支「虛擬工程團隊」

- gstack = Y Combinator 總裁 Garry Tan 開源的 Claude Code 設定檔
- 核心 claim：一個人加上這套工具，可以像 20 人的團隊一樣出貨
- 本質：一堆 Markdown 檔案組成的 slash commands，每個對應一個「虛擬角色」

### 代表性 Slash Commands

| Command | 角色 | 功能 |
|---------|------|------|
| `/office-hours` | YC Office Hours | 用 6 個「逼問」問題重構產品想法，挑戰假設與定位 |
| `/plan-eng-review` | 工程主管 | 鎖定架構、資料流、diagram 與邊界情況，在寫 code 前對齊技術設計 |
| `/cso` | 資安長 | 跑 OWASP Top 10 + STRIDE 威脅模型，主動找出安全漏洞 |
| `/ship` | Release 工程師 | 同步 main、跑測試、push、開 PR，全自動化交付流程 |

### gstack 揭示的三個核心

1. **代理人是「工作流程」而不是「全能 AI」**
   - gstack 整體流程：Think → Plan → Build → Review → Test → Ship → Reflect
   - 每個 skill 嚴格負責一個階段，下一個 skill 讀前一個 skill 的輸出
   - 教科書等級的 Pipeline + Context Passing

2. **Skills 比 Sub-Agents 更好用（通常）**
   - gstack 的 23 個 skill 都是純 Markdown，沒有一個是會自主迴圈的 agent
   - 真正的智慧來自**工作流程編排（workflow orchestration）**，而不是 agent 自主性

3. **標準化代理人可讀文件**
   - `AGENTS.md`、`CLAUDE.md`、`SKILL.md` 是 2025–2026 年浮現的代理人可讀文件標準
   - 你的 repo 除了給人看的 README，還要有給 Agent 看的 `CLAUDE.md`

> Context Engineering 的核心：代理人看得到什麼、看不到什麼，才是系統的主要工程槓桿。

### autoresearch vs gstack：兩個維度的對照

| 維度 | autoresearch | gstack |
|------|-------------|--------|
| 時間軸 | 縱向迭代（同一任務跑 100 次） | 橫向階段（think → plan → ship） |
| 核心抽象 | Loop + Verifier | Pipeline + Context |
| Agent 角色 | 一個 agent 通宵優化 | 23 個 skill 依序協作 |
| Harness 類型 | 固定 prepare.py + 固定 5-min | 固定 skill order + 固定 context 傳遞 |
| 適合學習 | Verifiability Mindset、Self-improving loop | Skill Design、Workflow Orchestration |

## PART 3｜為什麼 CSer 應該從這兩個套件入手？

### 三個理由

1. **建立「系統視角」而不是「套件使用者」視角**
   - 常見陷阱：看到 paper 附的 repo 就直接 pip install，把 LangChain 當標準答案
   - 讀 autoresearch 的 100 行 program.md，比讀 LangGraph 的 10,000 行原始碼更能理解 Agent 怎麼運作

2. **寫論文時必須解釋清楚「為什麼這樣設計」**
   - Reviewer 會問：你的 verification criterion 是什麼？你如何防止 reward hacking？你的 Harness 邊界在哪？
   - 用黑盒子框架，這些問題都答不出來

3. **研究工作本身越來越像 autoresearch 這個 repo**
   - Karpathy 的未來預言：研究是 AI agent 群的事情，方向是真的

### 課程核心概念對應

| 概念 | 具體對應 |
|------|---------|
| Spec-Driven Development | program.md 就是 spec（autoresearch） |
| Eval-Driven Development | val_bpb 就是評估（autoresearch） |
| Harness Engineering | prepare.py 防改 + skill 順序鎖定（兩者皆有） |
| Verifiability Mindset | 5-min budget + 固定 eval（autoresearch） |
| Snowball Development | 通宵 100 次迭代（autoresearch） |
| Context Engineering | skill 間的 context 傳遞（gstack） |

### 結語

- **Agentic 系統的使用者**：直接裝 OpenClaw 就好。強大的框架會幫你處理一切——直到出錯的那一天。
- **Agentic 系統的工程師（甚至研究者）**：請從 autoresearch 和 gstack 開始——小到你可以完整讀完，然後真正理解。

> 代理人時代最稀缺的能力，不是「會用 AI 的人」，而是「能設計 AI 協作系統的人」。知其然而知其所以然——這句話從來沒有在 Agentic 時代失效過，反而變得更重要了。

## PART 4｜設計你自己的 Agentic Flow（五步移植法）

### Step 1：找到你的問題
有沒有「反覆做、每次都想做得更好、但沒有時間」的任務？
- 論文文獻回顧（找論文 → 整理 → 找缺口 → 再找）
- 程式碼重構（重構 → 測試 → 找瓶頸 → 再重構）
- 競爭對手分析（搜尋 → 整理 → 找缺口 → 補資訊）
- 教材設計（寫草稿 → 自評清晰度 → 修改）

### Step 2：設計你的 val_bpb
品質指標必須是：可量化、與目標高度相關、可自動計算。
- 論文回顧：相關論文覆蓋率 ≥ X 篇、時間跨度 ≥ Y 年
- 程式碼重構：測試覆蓋率 ≥ 80%、執行時間 < X ms
- 競爭分析：必填欄位完成率 = 100%、資料來源 ≤ 3 個月

### Step 3：設計迴圈結構
```
SETUP → ACTION（Agent 每輪執行什麼）
     → EVALUATE（怎麼計算分數）
     → DECIDE（達標結束 / 找弱點補強）
     → TERMINATE（達標或達輪數上限）
```

### Step 4：設計 program.md
Agent 的指令文件必須包含：
- 目標（量化指標）
- 每輪動作（具體行動）
- 終止條件（明確）
- 輸出格式（具體規範）

### Step 5：定義 Harness 邊界
- **固定層**：評分框架、資料來源 API、輸出格式模板
- **行動空間**：搜尋策略、寫作內容、迭代優先級（優先補強哪個維度）

## PART 5｜AI 正在抹掉職缺嗎？——數據背後的職涯轉型真相

> 從 Stanford SDEL「煤礦金絲雀」研究看初階白領的結構性崩塌

### 三個衝擊數據

| 數據 | 來源 | 說明 |
|------|------|------|
| **13–16%** | Stanford SDEL (2025) | 22–25 歲在 AI 高暴露職業中，相對就業下降幅度 |
| **~50%** | SignalFire (2025) | 前 15 大科技公司對工作經驗 <1 年者的聘用數，2019–2024 年降幅 |
| **27.5%** | U.S. BLS / IEEE Spectrum | 美國「Programmers」整體就業 2023–2025 年降幅（「Software Developers」僅降 0.3%） |

> AI 不是在「裁員」——而是在系統性地「不請人」。

### 研究來源：「煤礦裡的金絲雀」

- **標題**：Canaries in the Coal Mine? Six Facts about the Recent Employment Effects of Artificial Intelligence
- **作者**：Erik Brynjolfsson、Bharat Chandar、Ruyu Chen（Stanford Digital Economy Lab）
- **數據基礎**：ADP 工資單數據，涵蓋數百萬勞工，月度更新至 2025 年 9 月
- 2026 年補充論文：顯著效應從 **2024 年**開始，而非 2022 年

**方法論嚴謹性：**
- 使用 firm-time fixed effects，排除經濟週期與利率影響
- 控制科技業集中度與遠距工作可能造成的偏差
- 結合 Eloundou/Tomlinson AI 暴露度量，進行固定效應迴歸

### 六大核心發現

1. **年輕暴露者就業下降**：22–25 歲在高 AI 暴露職業的相對就業下降 13–16%
2. **年長勞工不受衝擊**：同一職業中，經驗豐富者就業持平甚至增長
3. **調整透過就業數、非薪資**：企業並沒有降薪，而是不回補空缺——靜悄悄地移除入口
4. **自動化 > 增強**：下降集中在 AI 用於「取代」而非「輔助」的任務類型
5. **結果對多種替代解釋穩健**：排除科技業、遠距工作等因素後仍顯著
6. **結構性而非週期性**：非利率、非景氣循環造成——這是長期趨勢的起點

### 多個獨立數據源指向同一結論

| 來源 | 數據 |
|------|------|
| SignalFire 2025 | 大科技公司新鮮人聘用下降 50%（2019–2024） |
| EY 報告 2025 | 印度 IT 服務業初階職位減少 20–25% |
| LinkedIn / Indeed / Eures | 歐洲主要國家 2024 年初階科技職位下降 35% |
| WEF 未來就業 2025 | 2025–2030 年間，全球 22% 的現有職位將消失或大幅改造；39% 的核心技能將改變 |

### 為什麼是年輕人？隱性知識的經濟學

**顯性知識（Explicit Knowledge）**
- 教科書、規章、SOP、可標準化輸出
- 例：寫 CRUD API、撰寫標準化 marketing copy、基礎財報分析
- ❌ 這正是 LLM 在海量文本上訓練而來的**最強項**，也正好是剛畢業學生最引以為傲的能力

**隱性知識（Tacit Knowledge）**
- 直覺、情境判斷、人際博弈、身體化技能
- 例：如何在會議中讀出客戶的真實需求、如何在 code review 時看出架構問題
- ✅ 需要「在場」、「試錯」、「師徒傳承」——無法被編碼，因此難以被 AI 取代

> Polanyi (1966)：We can know more than we can tell.
> David Autor (2014) 將此命名為 Polanyi's Paradox——年輕人的比較優勢正好是顯性知識的新鮮庫存，而這正是 AI 最擅長替代的。

### 但等一下——另一個研究指向相反方向？

**Brynjolfsson, Li & Raymond（2023/2025, QJE）：Generative AI at Work**
- 5,172 位客服人員的真實實驗
- AI 輔助帶來 14–15% 平均生產力提升
- 新手與低技能者提升達 **34%**，資深員工幾乎無提升

**真正的解讀：**
- 對已被僱用的年輕人，AI 是最好的加速器
- 但僱用的門檻因為 AI 而被大幅提高
- 「入職越來越難，但一旦進去成長越來越快」——這是一個高度**兩極化**的賽局

### 職業階梯的斷裂：「吃掉自己的種子」

**傳統職涯模型（線性晉升）：**
新鮮人做搬磚 → 累積工具與流程 → 累積隱性知識 → 晉升中階管理者

**AI 時代的新現實：**
- AI 直接處理「搬磚工作」：簡單 coding、基礎分析、郵件草稿
- 企業短期獲利，省下人力成本
- 但長期災難：未來 10 年後，誰來當中階管理者？誰懂底層系統？

> Companies stopping junior hiring in 2025 are effectively eating their own seed corn. — Rezi.ai

IBM 的反向押注：2025 年宣布將年輕聘用人數擴大 3 倍，正是擔心中階人才池枯竭。

### IEEE Spectrum 微觀數據：不是所有職位都一樣慘

| 職位類型 | 2023–2025 就業變化 |
|---------|------------------|
| AI Engineers | +18% |
| Information Security Analysts | +15% |
| Software Developers | -0.3% |
| Programmers（行型） | **-27.5%** |

在「AI 生成程式碼」的時代，純執行型 coding 正在被貶值。真正值錢的是：系統設計（Architecture）、需求拆解（Problem Decomposition）、安全性判斷（Security Judgement）

### 「向下平權」的陷阱：生產力提升的分配不均

- 表面上：AI 壓縮了技能差距，看起來像民主化的勝利
- 實際上：**超額報酬集中在**
  - 算力持有者：NVIDIA、大型雲端業者
  - 模型持有者：Anthropic、OpenAI、Google
  - 戰略制定者：能把 AI 組織成價值系統的人

**「80 分的通貨膨脹」**：如果你的工作可以被 AI 生成 80 分，你的薪資就會趨近於「80 分的通貨膨脹後市場價」。

> 警語：別把「我會用 AI」當成護城河。每個人都會用。真正的護城河是 AI 無法輕易複製的判斷力與信任關係。

### Automation vs. Augmentation：不只是哲學問題，是飯碗問題

**Anthropic Economic Index 觀察：**
- Claude.ai 上：Augmentation 52% > Automation 45%（2025 年底）
- API 流量：Automation 主導，且比重持續上升——企業級 AI 使用正往純自動化移動

**你的個人選擇題：**
- ❌ Automation 路線：把工作丟給 AI，工作邊界縮小 → 下一個被裁的就是你
- ✅ Augmentation 路線：用 AI 擴張能力邊界 → 一個人活成一支軍隊

### 職業 AI 暴露圖像：Anthropic Economic Index

資工系學生注意：「電腦與數學」類別，理論暴露度高達 **94.3%** → 你必須做出差異化。

### 薪資動態：「靜默調整」的真相

**學術研究共同發現：**
- Anthropic（2026）：未觀察到大規模失業
- Stanford SDEL：調整主要透過「就業（不補缺）」而非「薪資」——靜悄悄地發生
- ICLE 文獻回顧（2026）：截至 2025 年末，總體勞動市場尚無大規模位移，但初階集中衝擊已明確可見

**J-Curve 的警示（Brynjolfsson, Rock & Syverson, 2021）：**
- 新技術的紅利有滯後，但初期調整成本落在最弱勢族群身上
- 此刻正是你們畢業的 2026–2028——恰好是調整成本最集中的時間窗口

## PART 6｜應對策略：四個動作

### 動作一：把 AI 當「陪練」而非「代筆」

- ❌ 錯誤示範：把作業丟給 ChatGPT → 貼上繳交
- ✅ 正確示範：
  - Spec-Driven Development：先寫清楚規格 → 讓 AI 產出 → 你做驗證與 code review
  - 讓 AI 當苛刻的客戶：「假裝你是 PM，嚴格 review 我的 HW3 RAG 設計」
  - 讓 AI 當敵對評審：「把這份架構提案當論文 reviewer 來撕」
- 目標：訓練你的**品質控制權**與**邏輯敏銳度**

### 動作二：從「線性晉升」轉向「職業網格（Career Grid）」

- 線性晉升：CS → 後端工程師 → 資深 → 技術主管
- 職業網格：基礎 × 領域 × 寫作 × 研究 = 倍增價值

**AI 如何賦能網格策略：**

## PART 8｜回到本質：這是 OS 的 IPC 問題

### Agent 系統 ↔ 作業系統 完整對照表

| 作業系統概念 | Agent 世界的對應 |
|------------|----------------|
| Process | Agent |
| OS Kernel | Agent Runtime（Claude Code / Pi） |
| fork() + exec() | Spawn sub-agent |
| Pipe（stdin/stdout） | Sub-agent 委派、CLI agent |
| RPC | MCP 呼叫 |
| TCP Socket | WebSocket / HTTP |
| Signal | Hook events |
| System call | Tool call / MCP |
| Library（libc） | Skill / Extension |
| Network Service | Agent-to-Agent (A2A) |
| Context Switch | Session compaction |
| APT / Package Manager | Agent package ecosystem |

### Intra-Process vs Inter-Process

**Intra-Process（單一 process 內）：**
- OS：Function call、Shared memory、Thread synchronization
- Agent 對應：Tool call、Skill、Extension hook

**Inter-Process（跨 process）：**
- OS：Pipe / FIFO、File + lock、Socket (TCP/UDP)、RPC
- Agent 對應：Sub-agent（parent-child pipe）、Mailbox（file IPC）、Gateway（WebSocket hub）、A2A Protocol

### 本質問題永遠是這幾個

1. 誰在控制執行流程？（確定性程式碼 vs 概率性 LLM）
2. 資源怎麼共享／隔離？（context window、token budget）
3. 同步還是非同步？（tool call 要 block 嗎？）
4. 信任邊界在哪裡？（是否需要權限檢查？）
5. 失敗時怎麼恢復？（Session replay？Checkpoint？）

## PART 9｜四大架構分類

### 總覽

| 類別 | OS 對應 | 代表方案 | 本質 |
|------|---------|---------|------|
| 1. Skill | Library / Plugin | Claude Skills、Pi Skills | Agent 內部能力擴充 |
| 2. MCP | System Call / RPC | Anthropic MCP | Agent → 外部 Tool |
| 3. A2A | Network Service | Google A2A Protocol | Agent ↔ Agent 水平協作 |
| 4. CLI Agent | Shell Process | Pi、Claude Code CLI | Runtime 本身 + stdio IPC |

> 關鍵洞察：這四類不是競爭關係，是不同層的東西。一個 agent 可以同時用四種機制。

### 類別 1：Skill — Agent 的內部能力

- 定義：Agent 內部的能力擴充，住在你 repo 裡
- OS 類比：函式庫（library），如 libc、numpy
- 控制流：**Deterministic**——步驟寫死在 markdown / code 裡，不依賴 LLM 判斷
- 執行位置：Agent 本身的 process / loop，不額外開 process
- 擴充方式：放進 skills 目錄，runtime 自動發現並載入
- 典型例子：「如何做 code review」、「如何寫 commit message」

### 類別 2：MCP — Agent 呼叫外部工具

- 定義：Agent 透過標準協議呼叫外部 service（資料庫、SaaS、API）
- OS 類比：跨 process RPC，如 gRPC、dbus
- 控制流：LLM 決定何時呼叫，server 提供確定性回應
- 執行位置：MCP server 是另一個獨立 process（可遠端）
- 協議：JSON-RPC 2.0（over stdio 或 HTTP/SSE）
- 典型例子：Gmail MCP、Google Drive MCP、自家資料庫

> MCP 重要區分：MCP 是「垂直整合」（agent 向下連接 tool），和 A2A 的「水平協作」完全不同。

### 類別 3：A2A — Agent 之間水平協作

- 定義：獨立存在的 agent 之間、跨信任邊界的協作協議
- OS 類比：Network service + 認證（HTTPS + OAuth）
- 控制流：雙方各自自主，沒有「parent」主控關係
- 執行位置：跨機器、跨網路、跨組織
- 協議：JSON-RPC 2.0 + SSE + OAuth 2.0
- Discovery：`/.well-known/agent-card.json`

**MCP vs A2A 常被搞混：**
- MCP = Agent → Tool（垂直），OS 對應：System Call
- A2A = Agent ↔ Agent（水平），OS 對應：Network Service

### 類別 4：CLI Agent — Runtime 本身作為介面

- 定義：把 agent 整個包成一個 shell 指令，用 stdio / file 和外界溝通
- OS 類比：Shell process（grep、awk、jq）
- 控制流：外部 orchestrator（human 或 script）決定
- 協議：JSONL over stdio、或純 stdin/stdout 文字
- 典型例子：pi, claude（CLI mode）, aider

```bash
echo "task" | pi | jq .output | some-other-agent
```

### 四大架構的 Stack 位置

```
使用者 (User)
    │
    ▼
CLI 代理層（Pi/Claude Code 作為容器運行時）── 水平進程間 A2A ──▶ A2A（代理到代理）
    │                                                              進程間水平、網絡服務、對等網絡
    ├── 技能（進程內庫/插件）
    ├── 擴展/掛鉤（進程內，修改運行時行為）
    └──▶ MCP（模型上下文協議）
         進程間垂直、系統調用 RPC 到外部工具/服務
```

### 決策樹：我該用哪一類？

```
把可重複的工作流程包成能力？
    → 類別 1: Skill（agent 內部、deterministic）

操作外部系統（DB、SaaS、API）？
    → 類別 2: MCP（另起 process、標準 tool 介面）

跟「獨立存在」的 agent 協商（跨組織）？
    → 類別 3: A2A（跨網路、需 Agent Card、需認證）

把 agent 變成可 shell 組合的工具？
    → 類別 4: CLI Agent（stdio 介面、Unix 哲學）
```

**反模式提醒：**
- ❌ 用 MCP 做 A2A 該做的事（忽略信任邊界）
- ❌ 用 Skill 做 MCP 該做的事（把外部 API 硬塞進 prompt 裡）
- ❌ 用 Sub-agent 做 Skill 該做的事（明明 deterministic 卻開 LLM loop）

### 組合範例：Code Review Agent Team

| 類別 | 應用 |
|------|------|
| 類別 4：CLI Agent | 三個 pi instances：Scanner / Analyzer / Reporter |
| 類別 1：Skill | 各裝專業 skill：run-lint.md / severity-ranking.md / markdown-format.md |
| 類別 2：MCP | 連接 GitHub MCP（讀 PR diff）+ Slack MCP（送通知） |
| 類別 3：A2A | 對外提供服務，跨信任邊界協作 |

## PART 10｜Pi Framework：親手解構的教具

### 為什麼選 Pi？

| 工具 | 特性 |
|------|------|
| Claude Code | 完備、功能強、預設幫你做了所有選擇 |
| Cursor | IDE 整合、綁在編輯器裡 |
| AutoGen / LangGraph | Library 型、抽象層級高 |
| **Pi** | **極簡 CLI、刻意不做選擇** |

Pi 刻意沒有的功能：
- ❌ No sub-agents（「有很多種做法，自己 spawn」）
- ❌ No MCP（「用 extension 自己加」）
- ❌ No permission popup（「跑在 container 裡，或自己寫」）
- ✅ 只有 4 個原始 tool：read / write / edit / bash

> Pi 不是最好的架構。它是最好的教具。

### Pi 全貌：極簡 agent kernel + 五種擴充機制

核心（Pi Core）：
- LLM 迴圈
- Session 樹（JSONL 格式）
- 4 個內建工具：讀取、寫入、編輯、bash

五種擴充機制：
1. 提示詞模板（Markdown，可重複使用）
2. 技能（SKILL.md 格式，可執行）
3. 擴充功能（TypeScript 模組，修改行為）
4. 主題（JSON 設定，UI/顏色）
5. Pi 封裝包（package.json 綁定）

### Pi 的四種執行模式

| 模式 | 指令 | OS 類比 | 使用時機 |
|------|------|---------|---------|
| Interactive | `pi` | 互動式 shell | 人類坐在終端前操作 |
| Print | `pi -p "..."` | 一次性指令（ls、grep） | Script 裡呼叫、CI/CD |
| JSON | `pi --mode json` | 結構化輸出 | 下游程式解析事件流 |
| RPC | `pi --mode rpc` | Daemon process + IPC | 被另一個 process 長期驅動 |

SDK 補充：`@mariozechner/pi-coding-agent` npm 套件 → 可在 Node.js 裡 import 使用

### 五種擴充機制（複雜度階梯）

1. **Prompt Template（最簡）**
   - Markdown 檔，可重用的 prompt（帶參數）
   - OS 類比：Shell alias
   
2. **Skill**
   - SKILL.md + 支援檔，「如何做 X」的 spec
   - OS 類比：Library / plugin
   - AIASE 課程的核心格式
   
3. **Extension**
   - TypeScript module，改 runtime 行為（tool / hook / UI）
   - OS 類比：Kernel module
   - 威力強大，能 hook event、能替換 syscall

4. **Pi Package（最強）**
   - package.json 宣告，把以上四種打包分發
   - OS 類比：APT package

### Extension API 骨架

```typescript
export default function (pi: ExtensionAPI) {
  pi.registerTool({
    name: "deploy", ...
  });
  pi.registerCommand("stats", { ... });
  pi.on("tool_call", async (event, ctx) => {
    // intercept & modify
  });
}
```

Extension 能力：自訂 tool、Sub-agent 實作、Permission gate、客製 UI、MCP 整合、Git checkpointing

### Session Tree：Pi 的版本控制

| Pi 功能 | CS 概念 |
|---------|---------|
| Session tree | Git commit graph |
| /fork | git branch |
| /tree | git log --graph |
| /compact | Git shallow clone / snapshot |

- Session tree = 「agent context 的版本控制」，所有 checkpoint、錯誤嘗試、平行探索都保留得住
- 這是 Verifiability Mindset 的基礎建設：任何 agent 決策都可以 replay

### Context Files 與 Skill 標準

**啟動時自動載入順序：**
```
~/.pi/agent/AGENTS.md    # 全域
../../AGENTS.md          # 從 cwd 上找
../AGENTS.md
./AGENTS.md              # 專案
```

也相容 `CLAUDE.md`——這是 2025–2026 年業界正在形成的 convention。

**Skill 格式：** Pi 遵循 Agent Skills 標準（agentskills.io）。Skill = 一個資料夾，內含 `SKILL.md` + 支援檔。

### Message Queue：Pi 的併發模型

| 按鍵 | 行為 | OS 類比 | 用途 |
|------|------|---------|------|
| Enter | Steering 訊息，現在的 tool 跑完就插隊 | SIGINT + 新指令 | 即時方向修正 |
| Alt+Enter | Follow-up 訊息，等 agent 整個完成才執行 | 排進 job queue 尾端 | 非緊急的後續指令 |
| Escape | 中斷 + 把未送出訊息還給 editor | SIGKILL + 狀態還原 | 緊急停止 |

背後是 **producer-consumer pattern + priority queue**（steering > follow-up）。

## PART 11｜AgentOpt：你的 Agent 管線，選錯模型了嗎？

> Columbia 大學 DAPLab 的 AgentOpt 論文，用 9 個模型、81 種組合的大規模實驗，顛覆了一個廣為流傳的假設——「最強的模型放在最重要的位置，就會得到最好的結果。」

### 最反直覺的實驗結果

測試基準：HotpotQA（多跳問答）

| 組合 | 準確率 |
|------|-------|
| 🏆 Ministral 8B（Planner）+ Claude Opus（Solver） | **74.27%** |
| ❌ Claude Opus（Planner）+ 任何 Solver | **31.71%**（倒數 11 名全是 Opus 當 Planner） |

差距 **42 個百分點**，贏的那組 Planner 是 Opus 價格的**三十三分之一**。

### 為什麼最強模型反而最差？

**Opus 太強了，強到不願意委派任務。**

論文團隊分析執行日誌發現：在 9 種 Opus 當 Planner 的組合中，有 7 種出現了**直接嘗試自己回答**，而不是把問題分解後委派給 Solver。

### 跨基準的系統性發現

| 基準 | 最佳組合 | 最佳準確率 | 關鍵洞察 |
|------|---------|-----------|---------|
| HotpotQA（多跳問答） | Ministral 8B + Opus | 74.27% | Planner 需要「守規矩」，不是「聰明」 |
| MathQA（數學推理） | Opus（Answerer）+ Haiku 4.5（Critic） | 98.84% | Critic 選什麼模型幾乎沒差，9 個 Critic 差距僅 2.9% |
| GPQA Diamond（研究生科學） | Opus 單獨 | 74.75% | 唯一「用最強模型就對了」的基準 |
| BFCL v3（多輪函數呼叫） | 三個模型並列 70% | 70% | 同分時，選便宜的就對了（成本差 32 倍） |

### 成本差距：「直接用最好的模型」多浪費？

| 基準 | 最佳 vs 最差組合的成本差 |
|------|----------------------|
| HotpotQA | 21 倍 |
| MathQA | **118 倍**（Opus+Opus vs Ministral+Haiku，準確率相近） |
| BFCL | 32 倍 |

### 為什麼你不能靠直覺選模型組合？

- 論文測試了 8 種搜索演算法，其中 LM Proposal 讓 GPT-4.1 來「推薦」最佳組合
- 結果：HotpotQA 上僅 34%，遠低於最佳的 74.27%

**🏆 贏家：Arm Elimination（基於信賴區間的逐輪淘汰策略）**
- 保持接近暴力搜索準確率的同時，節省 **40–60%** 的搜索預算

> Pipeline 中的角色交互是非線性的——你不能從「A 模型在 benchmark X 上表現好」推導出「A 模型在 pipeline 角色 Y 上表現好」。

### Anthropic 自己也在修正這個思路：Advisor Tool

| 階段 | 做法 |
|------|------|
| 過去的做法 | Opus 當主腦（Planner），便宜模型當手腳（Executor） |
| Advisor Tool 的做法 | Haiku / Sonnet 跑主迴圈，Opus 退居幕後當「顧問」，只在關鍵時刻被呼叫 |

- Haiku 搭配 Opus 顧問的表現大幅超過 Haiku 單獨運行，而總成本仍低於直接用 Sonnet
- 容錯設計：若顧問呼叫失敗，執行者不會中止

### 任務類型決定最佳分工

| 任務類型 | 最佳分工策略 |
|---------|-----------|
| 需要工具使用（如 HotpotQA） | 弱模型當 Planner 可能更好（不會「搶活幹」） |
| 純推理任務（如 GPQA） | 強模型獨挑就行 |
| 需要批判審查（如 MathQA） | Critic 角色的模型選擇影響不大，不值得花大錢 |
| 多步驟長程任務（如 SWE-bench） | 便宜模型跑主迴圈 + 強模型按需諮詢 |

### 你現在可以做什麼？

1. **審計你的 Agent 管線成本**——找出「假高價值」角色
2. **警惕「能力過剩」的陷阱**——檢查有沒有哪個角色太聰明，聰明到繞過了它該遵守的流程
3. **用系統化方法找最佳組合**——AgentOpt 是開源的，支援 LangGraph、AutoGen、CrewAI
4. **別相信直覺，包括 AI 的直覺**——GPT-4.1 在 HotpotQA 上推薦的組合只有 34% 準確率

### 一句話總結

> Agent 管線優化的單位不是「單一模型的能力」，而是「模型組合在特定任務上的匹配度」。用錯組合，你可能花 32 倍的錢，拿到一半的效果。
